# 全栈开发学习路线图（一年版）

> 目标：以「播客应用」为实战项目，从 Web 前端工程师转型为能独立交付产品的全栈开发工程师。
> 适用对象：已熟练 Vue3 + TypeScript，Node.js 经验较少，无数据库与后端项目经验。
> 时间投入：严格每天 2h，全年约 728 小时。

---

## 一、核心目标

一年后能够独立完成以下工作：

1. 设计并实现一个完整的全栈应用（前端 + 后端 + 数据库）。
2. 完成用户鉴权、RESTful API 设计、数据库建模、测试、部署上线。
3. 掌握缓存、消息队列、日志、监控、CI/CD 等生产环境必备技能。
4. 具备面试全栈岗位所需的算法、网络、系统设计基础。
5. 产出一个可写进简历、可演示的高质量开源项目。

---

## 二、技术栈选型

| 层级 | 技术 |
|------|------|
| 前端 | Vue3 + TypeScript + Vite + Pinia + Vue Router + Axios |
| 后端 | Node.js + TypeScript + NestJS |
| 数据库 | PostgreSQL（主数据库）+ Redis（缓存/会话/队列） |
| ORM | Prisma |
| 鉴权 | Passport.js + JWT + OAuth 2.0（GitHub） |
| 测试 | Vitest + Supertest |
| 部署 | Docker + Docker Compose + Nginx + GitHub Actions |
| 监控 | Sentry + Prometheus + Grafana |
| 队列 | BullMQ |
| 对象存储 | 阿里云 OSS / 腾讯云 COS / MinIO（本地开发） |
| 搜索 | PostgreSQL 全文检索 / Meilisearch |

**选型理由：**

- 优先复用 Vue3/TS 经验，降低语言切换成本。
- NestJS 结构清晰，便于理解企业级后端架构。
- PostgreSQL 功能全面，对 JSON、全文搜索、事务支持友好。
- Docker + GitHub Actions 是当前最主流的 DevOps 入门组合。

## 三、实战项目：播客应用

### 3.1 功能范围

- **用户系统**：注册、登录、JWT 鉴权、OAuth 登录、个人资料。
- **内容系统**：节目上传、封面/音频管理、分类、标签、搜索。
- **互动系统**：订阅播客、收藏单集、播放进度、评论。
- **后台管理**：节目审核、用户管理、数据看板。
- **通知系统**：新节目推送、邮件通知。
- **播放体验**：前端音频播放器、播放历史。

### 3.2 核心数据表

- `users`：用户
- `podcasts`：播客频道
- `episodes`：单集
- `categories`：分类
- `subscriptions`：订阅关系
- `favorites`：收藏关系
- `play_histories`：播放进度
- `comments`：评论
- `uploads`：文件上传记录

---

## 四、学习原则

1. **项目驱动，用中学**：每个知识点必须落在播客应用里。
2. **先跑通，再优化**：先实现功能，再考虑性能、安全、可观测性。
3. **每周有产出**：代码必须提交到 GitHub，不要只看不练。
4. **重视基础概念**：框架会变，HTTP、数据库、网络、并发模型不会变。
5. **及时复盘**：每阶段结束写总结，记录踩坑和收获。

## 五、阶段计划

### 阶段一：后端基础与数据库（第 1-8 周）

**阶段目标：** 能用 Node.js + PostgreSQL 实现播客应用的 RESTful CRUD。

| 周次 | 主题 | 核心任务 | 产出物 |
|------|------|----------|--------|
| 第 1 周 | TypeScript + Node.js 热身 | 配置 TS 环境；复习 Promise/async-await/Event Loop；用原生 `http` 写最小 JSON 服务 | 最小 HTTP 服务 |
| 第 2 周 | Express 框架入门 | 路由、中间件、请求生命周期；用 Express 实现内存版播客 CRUD | Express 播客 CRUD |
| 第 3 周 | PostgreSQL 基础 | 安装 PG；学习表、类型、约束、CRUD、JOIN、聚合、事务 | SQL 练习脚本 |
| 第 4 周 | 数据库设计 | 设计播客应用 ER 图；建表 SQL；理解索引、软删除、外键 | 数据库设计文档 |
| 第 5 周 | Prisma ORM | 配置 Prisma；迁移；把内存 CRUD 改为数据库版 | Prisma 版 CRUD |
| 第 6 周 | 复杂关系 | 实现 subscriptions / favorites / play_histories 多对多关系 | 核心业务模型 |
| 第 7 周 | API 设计规范 | RESTful 规范；统一响应格式；Zod 入参校验；分页/排序/过滤 | API 规范文档 |
| 第 8 周 | 阶段里程碑 | 完成节目列表、详情、分类筛选；整理 Postman Collection | 本地可运行的基础 API |

**阶段验收：** 不依赖前端，用 Postman 能完成播客节目的完整 CRUD 和分类查询。

---

### 阶段二：鉴权、测试与业务闭环（第 9-15 周）

**阶段目标：** 完成用户鉴权体系，核心业务流程跑通，具备基础测试覆盖。

| 周次 | 主题 | 核心任务 | 产出物 |
|------|------|----------|--------|
| 第 9 周 | Session + Cookie | express-session + PostgreSQL 存 session；登录态中间件 | Session 版登录 |
| 第 10 周 | JWT 鉴权 | Access Token + Refresh Token；受保护路由；token 刷新 | JWT 登录体系 |
| 第 11 周 | OAuth 第三方登录 | Passport.js + GitHub OAuth；账号绑定逻辑 | GitHub 登录 |
| 第 12 周 | 后端测试 | Vitest 单元测试；Supertest 集成测试；测试数据库隔离 | 核心接口测试 |
| 第 13 周 | 用户互动功能 | 订阅/取消订阅、收藏/取消收藏、播放进度保存 | 互动 API |
| 第 14 周 | 搜索与评论 | 基础模糊搜索；评论 CRUD；Swagger 文档 | 搜索/评论 API |
| 第 15 周 | 阶段里程碑 | 后端 API v1 完成；补全测试；整理 API 文档 | 完整后端 v1 |

**阶段验收：** 用户可注册登录、订阅播客、收藏单集、发表评论，接口测试覆盖率 > 50%。

---

### 阶段三：前端联调与本地完整跑通（第 16-17 周）

**阶段目标：** 前后端联调完成，本地可完整使用播客应用。

| 周次 | 主题 | 核心任务 | 产出物 |
|------|------|----------|--------|
| 第 16 周 | 前端骨架 | Vite + Vue3 + Pinia + Vue Router；路由与状态设计；Axios 封装 | 前端项目骨架 |
| 第 17 周 | 核心页面 | 首页、节目列表、节目详情、登录/注册、个人中心 | 前端主要页面 |
| 第 18 周 | 前后端联调 | token 自动刷新；统一错误处理；播放进度同步；修复 bug | 本地可演示 v0.1 |

**阶段验收：** 本地启动前后端后，能完成注册、浏览、订阅、播放等完整操作。

---

### 阶段四：部署与基础设施（第 18-21 周）

**阶段目标：** 应用上线公网，具备自动部署能力。

| 周次 | 主题 | 核心任务 | 产出物 |
|------|------|----------|--------|
| 第 18 周 | Docker 化 | Dockerfile 多阶段构建；docker-compose 编排 app + db + redis | 本地一键启动 |
| 第 19 周 | Linux 与 Nginx | 云服务器基础操作；Nginx 反向代理；HTTPS 证书；域名解析 | 服务器可访问 |
| 第 20 周 | CI/CD | GitHub Actions：测试 → 构建镜像 → 推镜像 → SSH 部署 | 自动部署流水线 |
| 第 21 周 | 阶段里程碑 | 生产环境数据库迁移；线上冒烟测试；部署文档 | 公网可访问的播客应用 |

**阶段验收：** `git push` 后自动完成测试、构建、部署，公网可正常访问。

### 阶段五：缓存、队列与可观测性（第 22-27 周）

**阶段目标：** 应用具备生产环境兜底能力。

| 周次 | 主题 | 核心任务 | 产出物 |
|------|------|----------|--------|
| 第 22 周 | Redis 缓存 | Redis 基础；缓存热门节目列表；session 存 Redis | 缓存加持的接口 |
| 第 23 周 | 缓存策略 | 缓存更新/失效；防穿透/击穿/雪崩；HTTP 缓存头 | 稳定缓存方案 |
| 第 24 周 | 消息队列 | BullMQ + Redis；音频转码异步处理；邮件通知任务 | 异步任务系统 |

| 第 26 周 | 监控与报警 | Sentry 错误追踪；Prometheus + Grafana 基础指标 | 可观测大盘 |

**阶段验收：** 能说出系统当前 QPS、错误率、慢接口；异步任务失败可重试。

## 六、全年算法学习安排

算法不是最后才突击，而是全年持续：

- **第 1-17 周**：每周 3 小时，以「理解数据结构」为主，每周 2-3 题。
- **第 18-34 周**：每周 4 小时，按标签刷题，每周 3-4 题。
- **第 35-46 周**：每周 6 小时，专项突破 DP、回溯、图论。
- **第 47-52 周**：模拟面试，限时刷题。

**刷题顺序：** 数组 → 哈希 → 双指针 → 链表 → 栈 → 队列 → 树 → 递归 → 回溯 → 动态规划 → 贪心 → 图论基础。

## 九、风险与应对

| 风险 | 应对策略 |
|------|----------|
| 工作太忙，周计划完不成 | 允许某周只完成 60%，但不要连续两周掉队；把任务拆成更小模块 |
| 某个技术卡太久 | 最多卡 3 天，先跳过，记录问题，后续回头看 |
| 学到后面忘记前面 | 每阶段结束写总结；定期回顾 GitHub 提交 |
| 项目越做越大收不住 | 严格按功能范围做，砍掉非核心功能 |
| 面试时不会讲项目 | 从第 20 周开始准备「项目介绍」话术，不断优化 |
