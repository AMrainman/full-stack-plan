# 今日核心知识点

## 1. 接口测试的四个基本场景

一个最小可用的 HTTP 接口测试应覆盖：

- **正常路径**：请求合法参数，返回 200/201 和预期数据。
- **404 路径**：请求不存在路由，返回 404。
- **400 路径**：请求参数非法，返回 400。
- **500 路径**：服务端内部异常，返回 500 且服务不崩溃。

为什么重要：只测正常路径会遗漏大量线上 bug，尤其是错误处理分支。

常见误区：
- 手动用浏览器点一遍就认为测试完成。
- 不写断言，只看「有没有返回」。

## 2. Demo 工程化整理

可复现的 demo 应包含：

- 入口文件：语义化命名，如 `podcast-server.ts`。
- `package.json`：声明依赖和运行脚本。
- `tsconfig.json`：明确编译目标和严格模式。
- 运行说明：每个文件顶部注释或单独 `README.md`。

为什么重要：一个月后回看，自己能立刻跑起来；面试时也能直接展示。

常见误区：
- 只保存一个 `.ts` 文件，忘记记录依赖版本。
- 路径写死，换台机器就无法运行。

## 3. 测试脚本的可重复运行

使用 bash 脚本时，注意：

- 启动后台服务后记录 PID。
- 测试结束后 `kill` 掉服务，避免端口占用。
- 使用 `set -e` 让脚本在断言失败时立即退出。

为什么重要：可重复运行是自动化测试的前提，也是 CI/CD 的基础。

常见误区：
- 手动启动服务后忘记关闭，下次运行报端口占用。
- 断言失败不退出，导致误判测试通过。
