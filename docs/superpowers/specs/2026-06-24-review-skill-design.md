# `/review` Skill 设计文档

> 设计日期：2026-06-24
> 适用仓库：full-stack-plan（播客应用全栈学习计划）
> 设计目标：为每天学习完成后提供交互式复盘，记录完成情况、卡点与掌握度

---

## 1. 背景与问题

`daily`/`today` skill 已经能自动生成每日学习计划（`tasks.md`、`review.md` 等），但每天实际学习结束后，用户需要：

- 记录哪些任务完成了、哪些没有。
- 标注未掌握的知识点，供第二天 `/daily` skill 读取并自适应调整。
- 把复盘结果结构化地写回当天的 `tasks.md` 和 `review.md`。

因此设计 `/review` skill，通过交互式提问完成每日复盘。

---

## 2. Skill 调用方式

- **文件位置**：`.claude/skills/review/SKILL.md`
- **命令格式**：

```text
/review                 # 自动定位最新 day-* 目录
/review <week> <day>    # 指定复盘第几周第几天
```

**示例**：

```text
/review        # 复盘最近一天
/review 1 1    # 复盘第 1 周第 1 天
/review 1 6    # 复盘第 1 周第 6 天
```

---

## 3. 目标目录定位

### 3.1 无参数 `/review`

1. 使用 `Bash` 运行：

   ```bash
   find daily -type d -name 'day-*' | sort -V | tail -1
   ```

2. 如果返回空，报错并停止：

   > 未找到任何已有计划，请先使用 `/daily 1 1` 生成第一天计划。

3. 从路径解析 `week` 和 `day`，例如 `daily/week-01/day-02` → week=1, day=2。

### 3.2 带参数 `/review <week> <day>`

1. 校验 `week` 和 `day` 为正整数。
2. 目标目录为 `daily/week-{ww}/day-{dd}/`，`{ww}` 和 `{dd}` 为两位补零。
3. 如果目录不存在，报错：

   > 未找到 daily/week-{ww}/day-{dd}/，请确认周/天参数正确。

### 3.3 目标文件

- `daily/week-{ww}/day-{dd}/tasks.md`
- `daily/week-{ww}/day-{dd}/review.md`

任一文件缺失，报错：

> 当天的 tasks.md 或 review.md 不存在，无法复盘。

---

## 4. 交互流程

找到目标目录并读取 `tasks.md` 和 `review.md` 后，按以下顺序交互：

### 4.1 逐项询问任务完成情况

读取 `tasks.md` 中所有 checkbox 任务（`- [ ]` 或 `- [x]`）。

对每个任务使用 `AskUserQuestion` 提问，选项：

- ✅ 已完成
- 🟡 部分完成
- ⏭️ 跳过（今天没做 / 不适合做）
- ❌ 未完成

> 示例提问：「任务：理解 TypeScript 编译流程与 `tsconfig.json` 核心字段。请选择完成情况。」

### 4.2 追问掌握度与卡点

对选择「🟡 部分完成」或「❌ 未完成」的任务，继续追问：

1. 掌握度选项：
   - 🟢 已掌握（只是时间不够）
   - 🟡 部分理解
   - 🔴 不太懂
2. 开放输入：请简述卡点或原因（可选，允许空）。

### 4.3 总体复盘问题

全部任务问完后，统一问两个开放问题：

1. 今天最大的收获是什么？
2. 有没有需要后续回顾的未解决疑问？

### 4.4 汇总预览与确认

向用户展示：

- 今日完成率
- 已完成任务数 / 总任务数
- 未完成任务列表
- 卡点与掌握度
- 今日最大收获
- 后续疑问

并提供选项：

- 确认写入
- 重新回答
- 取消

选择「重新回答」则回到 4.1；选择「取消」则不写入任何文件并结束。

---

## 5. 文件回写规则

用户选择「确认写入」后，同时修改 `tasks.md` 和 `review.md`。

### 5.1 `tasks.md`

- 已完成 → `- [x]`
- 部分完成 → `- [/]`；若渲染不支持，保持 `- [ ]` 并在行尾追加 `（部分完成）`
- 跳过 → 保持 `- [ ]`，行尾追加 `（跳过）`
- 未完成 → 保持 `- [ ]`，行尾追加 `（未完成）`

如果用户在 4.2 中输入了卡点，在该任务条目下方新增一行缩进说明：

```markdown
- [ ] 理解 TypeScript 编译流程与 `tsconfig.json` 核心字段（未完成）
  > 卡点：对 `moduleResolution` 和 `esModuleInterop` 的区别还不清楚
```

### 5.2 `review.md`

#### 5.2.1 更新「完成检查清单」

把 `review.md` 中「完成检查清单」的 checkbox 与用户回答同步：

- 已完成的任务 → `[x]`
- 未完成的任务 → `[ ]`

如果清单项与 `tasks.md` 的任务无法一一对应，则按用户总体完成情况更新。

#### 5.2.2 追加「今日实际复盘」

在 `review.md` 末尾追加一个区块：

```markdown
## 今日实际复盘（2026-06-24）

- 完成率：3 / 5（60%）

### 未完成任务

- [ ] 任务 A（部分完成）
  - 掌握度：🟡 部分理解
  - 卡点：xxx
- [ ] 任务 B（未完成）
  - 掌握度：🔴 不太懂
  - 卡点：xxx

### 今日最大收获

{用户输入}

### 后续疑问

{用户输入}
```

### 5.3 不回写 demo 文件

`/review` 只修改 markdown 状态记录，不动 `demo/` 目录里的代码文件。

## 6. 边界情况

| 场景 | 处理 |
|------|------|
| `daily/` 不存在或没有 `day-*` 目录 | 报错，提示先使用 `/daily 1 1` |
| 指定了 `week/day` 但目录不存在 | 报错，提示该日期没有生成计划 |
| 目标目录存在但 `tasks.md` 或 `review.md` 缺失 | 报错，提示无法复盘 |
| `tasks.md` 里没有 checkbox 任务 | 直接询问今日总体完成情况，按自由输入追加到 `review.md` |
| 用户在汇总预览时选择「重新回答」 | 回到 4.1 重新逐项询问 |
| 用户选择「取消」 | 不写入任何文件，直接结束 |

---

## 7. 实现方案

采用**纯 Prompt Skill**，不引入脚本或外部依赖：

- 在 `.claude/skills/review/SKILL.md` 中通过结构化 prompt 约束行为。
- 使用 `Read` 读取 `tasks.md` 和 `review.md`。
- 使用 `AskUserQuestion` 完成所有交互提问。
- 使用 `Edit` 或 `Write` 更新文件。
- 不自动调用 `/today` 或 `/daily`。

---

## 8. 与 `/daily` 的关系

- `/review` 不写新计划，只更新当天已有计划的状态。
- `/daily` 在生成第二天计划时，会读取前一天 `review.md` 中的掌握度标签和卡点，自适应调整任务难度和复习比例。
- `/review` 不自动触发 `/today`，保持职责单一。
