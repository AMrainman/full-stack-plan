# 全栈学习计划优化设计文档

## 一、背景与目标

- **仓库定位**：个人全栈学习仓库，以「播客应用」为实战项目。
- **目标受众**：已熟练 Vue3 + TypeScript，Node.js 与数据库经验较少的前端工程师。
- **时间约束**：严格每天 2 小时，全年约 **728 小时**（52 周 × 7 天 × 2h）。
- **优化范围**：
  1. `full-stack-plan.md` 的宏观阶段安排；
  2. 生成 `daily/week-XX/day-XX` 计划的 skill/模板；
  3. 在每日学习中嵌入 AI 辅助，提升学习效率。

## 二、当前问题

### 2.1 `full-stack-plan.md`

1. **总时间不匹配**：文档按「工作日 2h + 周末 5h」计算为 1040h，与实际 728h 相差 30%。
2. **阶段六过满**：8 周内需覆盖对象存储、全文搜索、WebSocket、后台管理、安全、网络并发、重构、收尾，在 2h/天 下难以充分完成。
3. **面试冲刺过长**：16 周占全年 30%，挤压项目深度与缓冲时间。
4. **缺少弹性缓冲**：任何一周掉队都会引发连锁延期。

### 2.2 `daily/week-01`

1. **内容重复**：day-01、day-02、day-04 都在反复配置 TypeScript 环境。
2. **只有 4 天计划**：第 1 周应有 14h，目前仅覆盖约 8h。
3. **日期字段错误**：day-02/03/04 的 `date` 均为 `2026-06-23`。
4. **未与周主题对齐**：按天独立生成，缺少周视角，容易偏离周目标。

## 三、优化原则

1. **总时间守恒**：以 728h 为硬约束，不增加隐性学习负担。
2. **前稳后弹**：基础阶段不压缩，把节省的时间变为缓冲。
3. **项目驱动**：每个知识点必须落在播客应用里。
4. **AI 作为加速器**：用 AI 压缩重复劳动，但不替代理解。
5. **每周有产出**：代码必须提交到 GitHub，不要只看不练。

## 四、全年阶段重新分配

| 阶段 | 原周数 | 优化后 | 新周数范围 | 说明 |
|------|--------|--------|------------|------|
| 一：后端基础 + 数据库 | 8 周 | 8 周 | 第 1-8 周 | 不压缩，基础必须打牢 |
| 二：鉴权 + 测试 + 业务闭环 | 7 周 | 7 周 | 第 9-15 周 | 不压缩 |
| 三：前端联调 | 3 周 | 2 周 | 第 16-17 周 | 学习者前端熟练，压缩 1 周 |
| 四：部署 + 基础设施 | 4 周 | 4 周 | 第 18-21 周 | 不压缩 |
| 五：缓存 + 队列 + 可观测性 | 6 周 | 6 周 | 第 22-27 周 | 不压缩 |
| 六：进阶功能 + 项目收尾 | 8 周 | 7 周 | 第 28-34 周 | 拆为「进阶 5 周 + 收尾 2 周」 |
| 七：面试冲刺 | 16 周 | 12 周 | 第 35-46 周 | 压缩 4 周 |
| 弹性/缓冲 | 0 周 | 6 周 | 第 47-52 周 | 分散到各阶段末尾 |
| **合计** | **52 周** | **52 周** | **52 周** | — |

### 4.1 缓冲分配建议

- 阶段一末尾：1 周（数据库设计容易卡）
- 阶段二末尾：1 周（鉴权与测试细节多）
- 阶段四末尾：1 周（部署上线容易延期）
- 阶段六末尾：1 周（收尾、文档、演示）
- 阶段七末尾：2 周（面试前总复习）

若某阶段提前完成，缓冲时间自动并入下一阶段或用于补前面未完成任务。

## 五、daily skill 生成策略

### 5.1 生成单位

- `week-XX` 是**主题边界**，不是**时间边界**。它标志「这一组任务连起来完成一个小 demo/产出物」，但不强制必须在 7 天内完成。
- 若某周太忙，可以自然延续到 `day-08`、`day-09` 等，skill 不强制 reset。
- 生成前先读取 `daily/week-XX/README.md`，理解本周主题与最终产出物。
- 生成时读取本周已生成的所有 `day-XX/tasks.md`，确保：
  1. 不重复已覆盖内容；
  2. 新任务是前一天任务的**自然延续**；
  3. 多天任务连起来能完成本周 demo。

### 5.1.1 连贯性规则

- 每一天任务必须明确写出「本任务完成后，demo/项目的最新状态是什么」。
- 下一天任务的起点必须是前一天结束时的项目状态。
- 不允许出现「今天学理论，明天突然换另一个无关主题」的断裂。
- 一个 `week-XX` 下的所有 day 最终应提交为一个完整的 Git commit（或一组相关 commit），对应一个小产出物。

### 5.2 每日任务结构

每个学习日统一为 4 段式，总时长 2h：

| 段落 | 时长 | 内容 |
|------|------|------|
| 理论学习 | 30-40min | 阅读文档/资料，理解一个核心概念 |
| 动手实践 | 40-60min | 在播客应用里写代码或做练习 |
| 验证/测试 | 15-20min | 跑通验收标准，确保产出可运行 |
| 复盘 | 10min | 写 review.md，记录收获与疑问 |

### 5.3 AI 辅助提示

每个任务后自动附加一行：

> AI 可以帮你：生成脚手架 / 解释报错 / 出练习题 / 审查代码 …（但修复和理解需自己完成）

### 5.4 周末/阶段性安排

- 不强制第 7 天必须收尾。若当周主题在自然第 7 天前后完成，建议：
  - **倒数第二天**：本周难点攻克 + 产出物整理。
  - **最后一天**：复盘 + 下周预习。
- 若第 7 天还没完成，继续生成 `day-08`、`day-09`……直到 demo 跑通再进入下一周主题。
- 每天仍保持 2h，不因周末加量。

### 5.5 弹性与自适应机制

#### 5.5.1 任务超时/未完成

- 若某天任务超时，学习者标记为「未完成」。
- skill 在次日自动生成「续接任务」，优先完成昨日遗留，再推进新内容。
- 若当周主题未在 7 天内完成，允许自然延续到 `day-08`、`day-09` 等，skill 继续按连贯性生成。

#### 5.5.2 掌握度驱动的复习

学习者在 `review.md` 中用以下标签标记前一天知识点的掌握度：

| 标签 | 含义 | skill 次日行为 |
|------|------|----------------|
| `🟢 已掌握` | 理解并能独立复现 | 正常推进新内容 |
| `🟡 部分理解` | 能看懂但不够熟练 | 今日 50% 时间做变式练习，50% 推进新内容 |
| `🔴 不太懂` | 概念模糊或跑不通 | 今日暂停新内容，专门做深度复习 |

**复习任务设计原则**：

- 不重复昨天的同一道题/同一段代码，而是生成**变式练习**。
- 用更简单的场景切入：例如昨天是「播客列表接口」，今天复习就用「单个节目详情接口」。
- 要求学习者用自己的话解释概念，并在 `review.md` 写「我现在的理解是……」。
- AI 辅助点：AI 根据昨天的卡点生成 3 个由浅入深的追问或练习。

**示例**：

- 昨天学 Event Loop，标记 `🔴 不太懂`。
- 今天任务不是重看文档，而是：
  1. 用手绘/文字描述 Event Loop 一轮执行流程；
  2. 给一段包含 `setTimeout`、`Promise`、`async/await` 的代码，预测输出；
  3. 把昨天的 HTTP 服务加一段异步逻辑，观察请求处理顺序。

- 每周日做周度 review，但 review 不强制清零；未完成项继续以「本周主题」推进。

## 六、week-01 示例（TypeScript + Node.js 热身）

第 1 周目标：搭建 TypeScript + Node.js 开发环境，理解异步编程核心概念，能写出可运行的原生 HTTP 服务。

> 注：下表示例按每天 2h、7 天完成设计。若某几天没空，任务会自然延续到 `day-08`、`day-09`……最终仍是同一个 demo 的逐步构建。

| 天 | 主题 | 核心任务 | 累计产出 | AI 辅助点 |
|---|------|---------|----------|----------|
| Day 1 | Node.js 运行环境 | 安装 Node、理解 npm/pnpm、运行第一个 `.ts` 文件 | 可运行的 TS 工程骨架 | AI 生成最小 `tsconfig.json` 并解释每个字段 |
| Day 2 | Promise / async / await | 手写 Promise 链、错误处理、`Promise.all` vs 循环 | `demo/promise-chain.ts` 可运行 | AI 出 3 道执行顺序题并讲解 |
| Day 3 | Event Loop | 读 Node.js 官方 Event Loop 文档、画执行图 | 能解释 macrotask/microtask 输出顺序 | AI 用播客场景解释 macrotask/microtask |
| Day 4 | 原生 HTTP 服务 | 用 `http` 模块写 `/health` 和 `/api/podcasts` | 最小 HTTP 服务可返回 JSON | AI 生成路由框架，学习者填处理逻辑 |
| Day 5 | 路由与错误处理 | 加 404/500 处理、异步路由、统一 JSON 响应 | 带错误处理的稳定服务 | AI 审查错误处理是否遗漏 |
| Day 6 | 本周难点攻克 | 手写一个带中间件风格的迷你路由；整理 demo | 可扩展的迷你路由 demo | AI 给出重构建议 |
| Day 7 | 复盘 + 预习 | 写 week-01 review；浏览 Express 下周主题 | week-01 review + 下周预习清单 | AI 根据 review 生成下周重点关注清单 |

### 6.1 主要变化

- 删除重复的 TS 环境配置，把 TS 压缩到 Day 1 的 30 分钟内。
- 把 HTTP 服务从 1 天扩展到 2 天，符合「先跑通再优化」原则。
- 周末不再空着，明确为难攻克和复盘。

## 七、AI 辅助学习集成

### 7.1 高优先级：直接省时间

1. **自动生成每日脚手架**：开新 demo 时生成 `package.json`、`tsconfig.json`、目录结构。
2. **报错解释与调试建议**：AI 解释报错原因和修复方向，学习者亲手修改。
3. **测试用例自动生成**：写完接口后 AI 生成 Supertest 骨架，学习者补断言。
4. **SQL 练习题生成**：根据播客 ER 图生成针对性 SQL 练习题。
5. **review.md 自动草稿**：根据当天任务和踩坑生成 review 初稿。

### 7.2 中优先级：提升理解深度

6. **代码审查反馈**：每周选 1-2 个文件让 AI 审查。
7. **API 设计评审**：检查 RESTful 规范、状态码、分页格式。
8. **概念串联讲解**：用播客应用具体场景解释抽象概念。
9. **自动架构图生成**：用文字描述生成 Mermaid 架构图。

### 7.3 低优先级：面试阶段再用

10. **模拟面试**：阶段七针对播客项目模拟技术面试。
11. **算法题思路提示**：卡住时给提示，不给答案。
12. **简历/项目描述优化**：阶段末根据 GitHub 提交历史优化描述。

### 7.4 AI 使用边界

| 可以让 AI 做 | 必须自己做 |
|-------------|-----------|
| 生成脚手架、模板、测试骨架 | 理解每一行代码为什么这样写 |
| 解释报错、给修复方向 | 亲手改完并验证 |
| 生成练习题和 review 草稿 | 总结真实收获和疑问 |
| 审查代码、指出问题 | 判断优先级并决定是否修改 |

## 八、风险与应对

| 风险 | 应对策略 |
|------|----------|
| 工作太忙，周计划完不成 | 每周日 review，必要时启用缓冲周；连续两周掉队则整体延后 |
| AI 辅助导致基础不牢 | 明确 AI 边界；每个知识点必须自己写一遍并跑通 |
| 阶段六进阶功能做不完 | 按「最小可用」原则，优先保证对象存储、搜索、后台管理；WebSocket 可延后 |
| 学到后面忘记前面 | 每阶段结束写总结；AI 根据 review 生成复习清单 |
| 面试时不会讲项目 | 阶段七前两周开始用 AI 模拟面试，不断优化项目介绍话术 |

## 九、验收标准

优化后的计划满足：

1. `full-stack-plan.md` 的总时间假设与实际 728h 一致。
2. 每个阶段有明确的产出物和验收标准。
3. 每周 7 天都有任务，且每天约 2h。
4. daily skill 生成任务时避免重复，并与周主题对齐。
5. 每个学习任务附带 AI 辅助提示，但不越界替代理解。
